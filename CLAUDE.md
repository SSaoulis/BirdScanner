# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

BirdScanner is a real-time bird detection and classification system designed for a Raspberry Pi with the Sony IMX500 AI Camera. The IMX500 runs object detection (YOLO11n) on-chip; detections are then passed to a ConvNeXt V2 Tiny ONNX model for bird species classification. The system must run on the Pi — `picamera2` and `libcamera` are not available on other platforms.

## How to develop

The development cycle is based on a worktree approach. Therefore there are a fixed set of rules to apply to develop: 

- Start by creating a worktree from main.
- ALWAYS pull latest changes from main onto the worktree before any files are read/analysed.
- Once a change is made, ALWAYS update the CLAUDE.md, and make a PR.

Use the .venv in the project root - this contains the linting/formatting tools and all required packages installed. 

## Commands

### Run the app (on Raspberry Pi only)
```bash
# from the repo root
python -m birdscanner.detector.main
```

Runtime behaviour is configured in `birdscanner/detector/config/config.py` (the former CLI args), not via
command-line flags. Edit the values on the module-level `config` instance:
- `multithread` — run classification on a background thread (prevents blocking the camera callback); defaults to `True`
- `object_duration_threshold` — seconds a track must be stable before classification fires (values `<= 0` floor the requirement to a single stable frame); defaults to `0.2`
- `debug` — enables `tracking` logger at DEBUG level; defaults to `True`
- `preview` — shows the camera preview window; defaults to `False`
- `save_video` — save a short mp4 clip per detection (best-frame still is always saved); defaults to `True`. `video_pre_roll_seconds` / `video_post_roll_seconds` size the clip (defaults `3.0` / `4.0`)
- plus `model`, `fps`, `bbox_normalization`, `bbox_order`, `threshold`, `ignore_dash_labels`, `preserve_aspect_ratio`, `labels`, `print_intrinsics`

### Run the app OFF the Pi (emulated camera, for local dev/debugging)
```bash
# needs assets/models/yolo11n.onnx (dev-only, out-of-band — see requirements.dev.txt)
FAKE_CAMERA_SOURCE=some_feeder_clip.mp4 python -m dev.run_emulated
# or, with no source set, it cycles the bundled tests/_test_images stills
python -m dev.run_emulated
```
This installs fake `picamera2`/`libcamera` modules (backed by a real YOLO11n ONNX object detector)
and then runs the real `main()`, so the whole detector — classifier, geomodel prior, SQLite writer,
control server, clip encoding — runs locally and writes real detections. Point the API at the same
`DB_PATH`/`IMAGE_DIR` to watch them in the UI. See the `dev/emulation` section below.

```bash
# All tests
pytest tests/

# Single test
pytest tests/ml/test_tracking.py::test_update_tracks_for_frame_increments_stability

# With coverage
pytest --cov=birdscanner tests/
```

### Type checking (mypy)

`mypy.ini` at the repo root configures mypy. Because everything is one package with
absolute imports, mypy runs in a **single pass**:

```bash
mypy birdscanner tools dev tests --check-untyped-defs
```

`mypy.ini` silences missing-import noise for the Pi-only native bindings (`libcamera`,
`picamera2`) and the untyped third-party libs (`onnxruntime`, `psutil`, `paramiko`, `PIL`
— Pillow ships no `py.typed` marker, `cv2` — OpenCV ships none either, plus `onnx` and
`matplotlib` which are used only by the dev-only scripts under `tools/`), so mypy reports
only genuine type errors in our own code.

### Linting (pylint, black)

Black and pylint need to be run for linting and formatting. 

Always run black first:

```bash
black birdscanner tools dev tests
```

Pylint rates the codebase **10.00/10** (exit 0). It runs in two invocations only because
the tests use a separate rcfile to relax pytest idioms — not for import resolution, which
is now uniform:

```bash
pylint birdscanner tools dev                 # source + dev tooling (root .pylintrc)
pylint tests --rcfile=tests/.pylintrc        # tests (test-tuned config)
```

  `tests/tools/test_build_species_reference.py`.
  - Parameter/value objects introduced for this: `ml/detection_utils.Box` (all box

## Architecture

### Detection pipeline (frame loop)

```
IMX500 on-chip inference
  └─ parse_detections()         # extract Detection objects from inference tensor
       └─ StableDetectionTracker.update_frame()   # IoU-based multi-frame tracking
            └─ process_detections() [picam2 pre_callback]
                 └─ ClassificationManager.process()
                      └─ process_single_detection_with_stable_tracks()  # new path
                           └─ BestFrameSelector.take(track_id)  # best frame seen across the track
                           └─ Classifier.classify()  # ConvNeXt V2 Tiny ONNX
                                └─ GeoPriorAdjuster.adjust()  # optional: Bayesian reweight by location prior
                                └─ save *clean* full image + 200×200 thumbnail to IMAGE_DIR/{species}/
                                └─ VideoRecorder.trigger()  → short mp4 clip (pre+post-roll, cropped ISP feed matching the still) to IMAGE_DIR/{species}/
                                └─ DetectionWriter.write()  → SQLite detections table (incl. normalized box + video_path)
```

The still image and thumbnail come from the **best frame observed across the track**
(highest YOLO detection confidence), not the arbitrary frame that triggered
classification — see `birdscanner/ml/best_frame.py`. Alongside the still, a short
**mp4 clip** is saved per detection (`birdscanner/detector/hardware/video_recorder.py`); the
clip finishes encoding a few seconds *after* the DB row is written, so its file (and
the `/video` endpoint) briefly 404s while the UI falls back to the still.

By default the **clip records the same cropped, ISP-processed `main` frame as the
still** (`config.video.full_fov = False`), so the clip matches the still exactly —
same detection crop, same aspect ratio, same image quality (ISP denoise/sharpening),
and browser-playable H.264. The recorder simply receives the cropped `main` frames
(no `video_frame_source` is wired), so its resolution is the `main` stream size
(`DEFAULT_LONG_SIDE = 1280` on its longer edge — bumped from 640 to make the clip/still
sharper; the real detail is still bounded by the crop region's own sensor-pixel size).

Setting **`config.video.full_fov = True`** reverts to the old behaviour: the **clip
shows the whole, uncropped field of view**. There is only one `ScalerCrop` per camera
and it applies to every processed stream, so the full-FOV clip is fed from the camera's
**raw** sensor stream (the full-FOV binned readout, unaffected by ScalerCrop) —
debayered + downscaled to RGB per frame by `birdscanner/detector/hardware/raw_frame.py`, wired
in via `PipelineContext.video_frame_source`. This full-FOV path is ISP-less (softer,
noisier, 4:3) and costs extra CMA to request the raw stream unpacked, which is why it
is off by default.

The saved full image is the **raw frame — no bounding box is drawn on it** (the box
is only drawn on the live preview stream). The detection box is persisted to the DB as
normalized `[0, 1]` coordinates (`box_x/y/w/h`) so the frontend can overlay it on the
clean image with an on/off toggle (see `frontend/src/components/Lightbox.tsx`).

### Package layout

The whole codebase is one package, `birdscanner/`, run via `python -m ...` with absolute
imports. It has four layered subpackages plus the consolidated assets and dev tooling:

```

              subpackages by concern:
                main.py, paths.py          entry point + data-file resolution (top level)

                                           video_recorder, camera_server

              tracking, classification, classification_pipeline, best_frame, geomodel)
  api/        FastAPI REST API + routers (was backend/)
  db/         SQLite persistence
assets/       labels/ (coco_labels.txt, geomodel labels + geomodel_classifier_map.json),
              models/ (classifier ONNX + class map), species_reference/ (data bank)
dev/          dev/test-only, NOT runtime and NOT shipped in the Docker images —
              the off-Pi IMX500 camera emulator (run_emulated + emulation/;

```

Import direction is one-way: `detector → ml → db` and `api → db`. `ml/` must never import
from `detector/` or `api/`.

### Key modules

**`birdscanner/ml/object_detection.py`** — core object detection only:
- `Detection` — bounding box + category + confidence; box is set after coordinate conversion via `imx500.convert_inference_coords`
- `parse_detections` — extracts `Detection` objects from the IMX500 inference tensor; `get_labels` filters the intrinsics label list
- `filter_excluded_detections(detections, labels, excluded)` — drops detections whose object-detection (COCO) class is in the exclude set (matched case-insensitively; out-of-range category indices are kept for the downstream guard). Called once per frame in `_run_capture_loop`, right after `parse_detections`, so excluded classes (e.g. the `"bench"` false positive that floods the tracker logs) never reach the tracker/drawing/classification. Returns the same list object when nothing is excluded (the no-op fast path). The exclude set is `config.excluded_classes`, editable live from the Settings page
- `last_detections` — module-global fallback returned when a frame yields no inference output

**`birdscanner/ml/detection_utils.py`** — stateless geometry/drawing helpers shared across the pipeline:
- `Box` — the box **value object** that owns all geometry (`.x2`/`.y2`/`.area`, `.iou(other)`, `.padded_square(img_w, img_h)`, `.crop(image)`, `.normalized(shape)`). Centralising the maths here is what keeps the callers under the `too-many-locals` limit. The module functions below are thin, tuple-based wrappers over it so existing call sites keep passing plain `(x, y, w, h)` tuples
- `iou` — Intersection-over-Union for two `(x, y, w, h)` boxes (delegates to `Box.iou`)
- `preprocess_roi` — expands a box to a padded square and crops the ROI (delegates to `Box.padded_square`/`.crop`)
- `draw_boxes(image, coords, detection, labels, classification=None)` — annotates a frame with a box + label + optional `(species, confidence)` classification result (grouped into one param to stay under `too-many-arguments`). Used only for the **live preview** stream now — it is no longer drawn onto the saved image (the box is persisted instead, see `normalized_box`)
- `normalized_box` — converts a `(x, y, w, h)` pixel box to fractions of the image dimensions, clamped to `[0, 1]` (delegates to `Box.normalized`); persisted with each detection so the UI can overlay the box on the clean saved image at any rendered size
- `save_thumbnail` — writes a 200×200 JPEG thumbnail
- `label_for_category` — bounds-checked label look-up; returns `None` when the class index is outside the label list. The IMX500 SSD model occasionally emits a spurious detection whose category index is out of range, so every `labels[category]` access goes through this helper. `draw_boxes` falls back to an `id:<n>` placeholder, and `process_detections` skips the detection entirely (logging a warning to the `tracking` logger) instead of crashing the camera callback with an `IndexError`

**`birdscanner/ml/tracking.py`** — multi-frame stability tracking:
- `StableDetectionTracker` — IoU-based tracker; a detection must match across `min_stable_frames` consecutive frames before `should_run_bird_classification_for_detection` returns `True`; each track is classified at most once (`mark_classified`)
- `StableTrack`, `match_detection_to_track`, `update_tracks_for_frame`, `should_classify_track` — the underlying pure-function tracking primitives (directly unit-tested in `tests/ml/test_tracking.py`)
- `stable_detection_tracker` — module-global default tracker instance

**`birdscanner/ml/classification_pipeline.py`** — classification orchestration + persistence:
- `PipelineContext` — the **parameter object** bundling every per-detection dependency (`classifier`, `tracker`, `classify_fn`, `detection_writer`, `best_frame_selector`, `record_fn`, `video_frame_fn`, the optional `video_frame_source` (the raw→full-FOV clip-frame extractor; see below), and the optional `geo_adjuster`) plus the two **live-editable save-side settings** (`save_confidence_threshold`, defaulting to `DEFAULT_SAVE_CONFIDENCE_THRESHOLD` = 0.4, and `ignore_species`, a lower-cased set); the Settings page mutates these fields on the running context via `settings_controller` (they live here rather than on `config` because `ml/` must not import `detector/`). `_should_persist(result, context)` gates saving on the confidence floor **and** the ignore list. Its `__post_init__` fills `tracker`/`classify_fn` from the module defaults. Built once in `main._build_manager` and shared by `ClassificationManager` **and** the processing functions, so neither has a long argument list. `Still` (frame + box) and `Classification` are small `NamedTuple` value objects that keep the persistence helpers under the argument/local limits. **`Classification`** carries the final `species`/`confidence` plus four optional fields (`classifier_species`, `classifier_confidence`, `geo_scores`, `classifier_scores`). `_predict_species` computes the classifier softmax **once** when the classifier supports it (guarded by `_classifier_probs`, which returns `None` for the bare-`object()` test fakes → fall back to the injectable `classify_fn` for a top-1-only result, `classifier_scores` stays `None`). From that single vector it always records `classifier_scores` (the classifier's own top-`CLASSIFIER_TOP_K` `[species, probability]` pairs via `classification.top_k_predictions`, JSON); then, when a `geo_adjuster` is wired, it runs `GeoPriorAdjuster.adjust` on the same vector so `species`/`confidence` are the **geomodel-corrected posterior**, `classifier_species`/`classifier_confidence` keep the classifier's own pre-adjustment pick, and `geo_scores` is the JSON top-K pre-normalised vector; with no adjuster `species`/`confidence` are the softmax argmax and those geomodel fields stay `None`. `_should_persist` gates on the **posterior** confidence
- `ClassificationManager(context, *, use_multithreading=False, queue_maxsize=0)` — wraps sync/async (threaded `Queue`) dispatch; in async mode items are dropped if the queue is **full** so the camera callback never blocks; all pipeline deps come from the injected `PipelineContext` (so `ml/` never imports picamera2). **The queue must be bounded for the drop to ever fire** — each queued item pins a full-resolution `main` frame (~5 MB at `DEFAULT_LONG_SIDE=1280`), so an *unbounded* queue (`queue_maxsize=0`, the constructor default) can never be "full", never drops, and grows without limit whenever the CPU classifier falls behind a busy feeder — leaking memory until RAM pressure stalls the camera pipeline (the capture loop wedges on `capture_metadata()` and `/capture` starts 503-ing while the separate control-server thread still answers). `build_manager` therefore passes an explicit bound (`gating.CLASSIFICATION_QUEUE_MAXSIZE = 32`, ~160 MB worst-case in-flight); dropping is safe because the `BestFrameSelector` retains each track's best frame independently, so a stable track still classifies from its best observed frame even if intermediate detections are dropped. Both the sync and async paths run through `_dispatch`, which **catches and logs any exception from processing a single detection** so one bad detection can never take down the pipeline: an unhandled exception in the async worker thread used to kill it permanently — after which every queued detection silently piled up unprocessed and **nothing was ever classified or saved again** (the symptom was tracks becoming stable but staying `species=None`, with no new DB rows / gallery entries). The async `_worker_loop` also `task_done()`s in a `finally` so the queue never wedges
- `process_detections` — picam2 `pre_callback` entry point; draws boxes, feeds every clean frame to `context.video_frame_fn` (the pre-roll buffer), offers each bird detection to the `context.best_frame_selector` (keyed by its track), and queues bird detections. The frame fed to the video buffer is the **full uncropped FOV**: when `context.video_frame_source` is wired it is called with the camera `request` to produce the debayered full-FOV frame (from the raw stream), falling back to the cropped `main` frame when unset or when the source returns `None`. The stills/thumbnails/classification still use the cropped `main` frame
- `process_single_detection_with_stable_tracks(item, context)` — the stable-track gating path (the only classification path; the old per-frame IoU-reuse path was removed). Decomposed into small helpers (`_best_still`, `_classify_track`, `_save_still_and_thumbnail`, `_start_clip`, `_persist_detection`) to stay under the complexity limits. At the stable trigger it takes the track's **best frame** from `best_frame_selector` (falling back to the trigger frame) and uses it for the classification ROI, the saved **raw** still (no box drawn), the thumbnail, and the persisted normalized box; `_persist_detection` builds a `DetectionRecord` and calls `DetectionWriter.write(record)`. It also calls `record_fn` to start a clip: `_start_clip` returns a `(video_path, no_video_reason)` pair, so `video_path` is persisted only when recording actually began and, when it didn't, `no_video_reason` records **why** (`"recorder_busy"` for a single-flight-declined trigger, `"disabled"` when no recorder is wired) — the UI reads this to explain the greyed-out Video toggle. **Skips classification for a degenerate (zero-area) detection box** whose ROI is empty (`roi.size == 0`) — feeding an empty array to the classifier raises inside PIL (`Image.fromarray`); the track is left unclassified so a later, non-degenerate frame can still classify it. This was the most common trigger of the worker-death bug above
- `setup_classifier`, `run_bird_classification`
- `IMAGE_DIR` — root directory for saved images, sourced from the `IMAGE_DIR` env var (defaults to `/home/stefan/Pictures/bird_detections`)

**`birdscanner/ml/best_frame.py`** — per-track best-frame selection (pure numpy, unit-tested in `tests/test_best_frame.py`):
- `BestFrameSelector` — thread-safe (`observe`/`take`/`discard`) store that keeps the single highest-scoring `(frame, box)` per `track_id` (≤1 frame retained per active track). Written on the camera thread (`observe`), read on the classifier thread (`take`), and freed when a track ends via the tracker's `on_track_deleted` callback (`discard`). The score is the detection's **YOLO confidence** (`detection.conf`) — zero extra compute. `BestCandidate` is the retained `frame`/`box`/`score` dataclass

**`birdscanner/ml/geomodel.py`** — BirdNET geomodel (spatio-temporal species prior), the geomodel↔classifier **label crosswalk**, **and** the runtime Bayesian update that reweights each classification by the location prior (unit-tested in `tests/ml/test_geomodel.py`):
- `load_labels` / `generate_grid_prediction` / `bayesian_update` — load the tab-separated geomodel label file (`id\tscientific\tcommon`, file order = output-index order), run the ONNX geomodel over all 48 weeks for a lat/lon to get per-species sigmoid priors, and Bayes-combine a prior with a likelihood
- **Runtime Bayesian update** (`week_of_year` / `build_prior_matrix` / `geomodel_posterior` / `GeoPriorAdjuster`) — applies the stored prior to a classifier softmax at inference time so a species that doesn't occur near the camera is demoted after a mis-classification. `geomodel_posterior(classifier_probs, prior)` computes `p(y|x,c) = p(y|x)·p(y|c) / Σ p(y'|x)·p(y'|c)` and returns **both** the normalised posterior **and** the *pre-normalised* product `p(y|x)·p(y|c)` (persisted per detection for debugging); it falls back to the classifier probs if the product sums to 0. `build_prior_matrix(priors, idx_to_class, *, floor=1e-4, unmapped_prior=1.0)` aligns the `{species: [48 floats]}` store to a `(48, n_classes)` matrix in classifier-index order — each stored prob is **floored** (absent species made *unlikely*, not *impossible*, so a genuine vagrant can still be reported) and classes with no crosswalk entry (e.g. the `Unknown` sentinel) get a **neutral** `1.0` (the update leaves them untouched). `week_of_year(when)` bins a date into `1..48` (four per month). `GeoPriorAdjuster(priors, idx_to_class)` precomputes that matrix once at startup and `.adjust(probs, week)` returns a `GeoAdjustment` (posterior argmax + confidence, the classifier's own pre-adjustment pick, and the top-K pre-normalised `[species, score]` pairs). Built by `detector/pipeline/gating.build_geo_adjuster` from the DB-stored prior; wired into `PipelineContext.geo_adjuster`
- `load_name_mapping` / `project_to_classifier` / `compute_classifier_priors` — the startup projection of the geomodel prior onto the classifier's classes. `load_name_mapping(path)` reads the curated `geomodel_classifier_map.json` (`{classifier_label: geomodel_common_name}`). `project_to_classifier(predictions, geomodel_labels, mapping)` takes the `(48, n_geomodel_species)` output of `generate_grid_prediction` and, for each `classifier_label -> geomodel_common_name`, slices that species' column into a 48-week vector, returning `{classifier_label: [48 floats]}` — many-to-one safe (shared columns) and it drops classifier classes whose geomodel species is absent; on a duplicate geomodel common name the first row wins (matching `build_name_mapping`). `compute_classifier_priors(model_path, labels_path, mapping_path, lat, lon)` ties inference + projection together and is the only piece here that loads the geomodel ONNX. This produces the `~706 species × 48 weeks` prior persisted to the DB on startup (see `detector/pipeline/geo_priors.py` + `db/geo_prior_store.py`)
- **The naming problem**: the geomodel emits ~12k species labels in **eBird/Clements** naming (Title Case) while the classifier's ~700 classes use **IOC-style** naming (sentence case, apostrophes/accents stripped). A naive exact-string match on the common name yields only ~31 hits; the two lists genuinely disagree on casing, punctuation, British `grey`↔American `gray`, **and** true synonyms (`Common Blackbird`↔`Eurasian Blackbird`)
- `normalize_common_name(name)` — folds case/diacritics/punctuation and unifies `grey`→`gray` into an alphanumeric key, so spelling-only differences collapse (resolves ~92%, 649/707). It does **not** bridge genuine cross-checklist synonyms — those need curation
- `build_name_mapping(geomodel_labels, classifier_labels, overrides=None)` — returns `(mapping, unmatched)`: `mapping` is **`classifier_label -> geomodel_common_name`** for every auto-matched class **plus** the hand-curated `overrides`; `unmatched` is the sorted classifier labels still without a counterpart. It is **classifier-keyed** (not geomodel-keyed) on purpose: the geomodel's per-species prior is projected *onto* the classifier's classes, and the relation is **many-to-one** — several classifier classes can share one geomodel species after an eBird lump (e.g. both `Common redpoll` **and** `Arctic redpoll` → `Redpoll`), which a geomodel-keyed dict cannot represent (the second write would clobber the first). First geomodel row wins on a normalised-key collision, so the map is stable across re-runs
- **Curation status (V3.0.3): 706/707 classes mapped, only `Unknown` (the non-bird sentinel) left.** Of the 58 that normalisation missed, 53 were confident cross-checklist renames resolved via the **scientific name** (the reliable join key — e.g. `Common blackbird`→`Eurasian Blackbird`, `Grey plover`→`Black-bellied Plover`, `Red-rumped swallow`→`European Red-rumped Swallow` (`Cecropis rufula`, the WP bird, **not** Eastern `daurica`), `Northern goshawk`→`Eurasian Goshawk`). The remaining **4 are geospatial proxies** where no exact species exists in the geomodel (lump/split): `Arctic redpoll`→`Redpoll` (eBird lumped it into `Acanthis flammea`), `Feas petrel`→`Cape Verde Petrel`, `Rock martin`→`Pale Crag-Martin` (the WP `Ptyonoprogne obsoleta`, not sub-Saharan `fuligula`), `Red-tailed wheatear`→`Persian Wheatear` — each chosen as the existing geomodel species whose lat/long/week occurrence best approximates the real bird. NB: the classifier's `class_to_idx.json` carries **no scientific names**, so this scientific-name resolution was done by hand; a future `class_to_idx` with scientific names could regenerate it automatically. NB2: the geomodel has a handful of **duplicate common names** (e.g. `Tyto alba` and `Tyto furcata` are both "American Barn Owl") — only `Barn owl` among our classes is affected; if the map ever needs to be robust to that, key its **values** by the geomodel's eBird species code (label-file column 1) instead of the common name
- The generated `assets/labels/geomodel_classifier_map.json` (a flat, sorted `{classifier_label: geomodel_common_name}` object) is **curated source** — a `!assets/labels/geomodel_classifier_map.json` negation keeps it tracked despite the repo-wide `*.json` ignore. Build/refresh it with `tools/build_geomodel_map.py` (below)

The `SpeciesCorrection` body's `allow_new` (default `False`) is always forwarded: when `True` it permits `species` to be a **brand-new label not in the classifier vocabulary** — the detector registers it as a **custom species** (see `birdscanner/db/custom_species.py`) and files the detection under it, so a bird the model was never trained on can still be recorded; when `False` an unknown label stays a 400 (the correction path's typo guard). Both the delete `_detector_error_detail`-style helper and settings' now share `birdscanner/api/detector_proxy.py:detector_error_detail(resp)` (the one piece of proxy logic worth centralising — pulling `{"error": …}` out of the detector's JSON error body; kept in one place so `pylint`'s `duplicate-code` stays quiet)
- `birdscanner/api/routers/images.py` — `GET /api/images/{id}/thumbnail`, `GET /api/images/{id}/full`, `GET /api/images/{id}/video` (mp4 clip; 404 when the row has no `video_path` or the file isn't on disk yet — the clip finishes encoding a few seconds after the row is written), `GET /api/images/download?ids=...` (chunked ZIP of the stills)
- `birdscanner/api/routers/system.py` — `GET /api/system` (CPU/mem/disk/temp/uptime via psutil)
- `birdscanner/api/routers/network.py` — network monitoring + on-demand speed test (prefix `/api/network`). **Passive usage graph**: a module-level `_Sampler` daemon thread (started at import, mirroring `system._BOOT_TIME`) reads the host NIC byte counters (`psutil.net_io_counters`) every `_SAMPLE_INTERVAL_SEC=3` s into a ~1 h ring buffer, deriving per-sample download/upload **kbps** from the counter delta (the first reading only seeds the baseline). `GET /api/network/history?range=5m|30m|1h` returns `{interval_sec, samples:[{t, rx_kbps, tx_kbps}]}` for the window (400 on an unknown range). Reading counters transfers nothing, so the sampler runs continuously at no bandwidth cost. **Active speed test**: `POST /api/network/speedtest` calls `run_speed_test()` which downloads `SPEEDTEST_DOWNLOAD_BYTES` (~1 MB) from and uploads `SPEEDTEST_UPLOAD_BYTES` (~256 KB) to Cloudflare's `__down`/`__up` endpoints (overridable via `SPEEDTEST_DOWN_URL`/`SPEEDTEST_UP_URL`) and reports measured Mbps each way + bytes transferred; 503 when the endpoint is unreachable. Payloads are deliberately small to spare the Pi's limited connection — the test is a manual button, **never polled**. `run_speed_test` is a module function so tests monkeypatch it without hitting the network
- `birdscanner/api/routers/species.py` — `GET /api/species` (list with counts, sorted by count desc), `GET /api/species/vocabulary` (the classifier's full species vocabulary for the correction picker), and `GET /api/species/expected` (the geomodel's "expected this week" list — see below). The API image ships **no** `assets/` (`Dockerfile.api` copies only `birdscanner/` + `frontend/dist`), so it cannot read the class map itself — `vocabulary` **proxies** the detector's `GET /labels` (the **union** of the classifier's classes and any user-added custom species, so newly-added labels appear in the picker) and 503s when the detector is unreachable (which is fine: corrections need the detector up anyway). Its 503 message is deliberately worded `"Species list unavailable: …"` (distinct from settings' `"Detector unavailable: …"`) so the two GET-proxy blocks don't trip `pylint`'s `duplicate-code`. `GET /api/species/expected?limit=` (default 6, cap 24 — small so the Dashboard band stays a compact glance) reads the DB-stored geomodel prior directly (the `geo_priors` / `geo_prior_meta` tables, mounted read-only) and returns the highest-scoring classifier species for the **current week** as `ExpectedThisWeek` (`{week, latitude|None, longitude|None, species:[{species, score}]}`, `score` = the raw occurrence probability [0,1]). The current week is computed by a **local** `_current_geo_week()` copy of `ml/geomodel.week_of_year` — the API must not import `ml.geomodel` (it imports `onnxruntime`, absent from the api image). When no location is configured (or the prior was never built) it returns a clean 200 with `latitude=longitude=None` and `species=[]`, which the Today page turns into a "set your location" prompt rather than an error. Powers the Dashboard's "In season · this week" panel
- `birdscanner/api/routers/stats.py` — aggregate **Statistics** endpoints (prefix `/api/stats`) powering the Statistics page. Every endpoint is a read-only SQL `GROUP BY` (so only aggregated buckets cross the wire, never raw rows), using SQLite `func.strftime` for time bucketing on the stored **naive-local** timestamps (bins are therefore local wall-clock). A small `StatsRange` `Depends` (mirroring `DetectionFilters`) applies the optional `from`/`to` window shared by the temporal endpoints. Endpoints: `GET /summary` (total / distinct-species / first+last timestamp / corrected-count), `GET /time-of-day` (a fine minute-of-day histogram — `bin_minutes` default 15 — plus optional `species`; the client circular-Gaussian-smooths it into a density curve, so raw rows never ship), `GET /activity` (per `(dow, hour)` counts; SQLite `%w` Sunday=0 is re-based to **Monday=0** in Python), `GET /timeline?interval=day|week&top=6` (per-interval counts split into the top-N species + an `"Other"` bucket, **plus** `distinct_species` per point — one query feeding both the stacked sightings chart and the diversity line), `GET /daily-window` (per-day earliest/latest minute-of-day), and `GET /first-sightings` (each species' earliest detection, oldest-first — intentionally **all-time**, no range, for the cumulative new-species curve). The most-common bar chart reuses the existing `GET /api/species`. **Gotcha fixed:** minute-of-day bucketing uses floor division (`//`) — SQLAlchemy's `/` renders Python *true* (float) division, which split one clock-minute bucket into spurious float groups. `func.count`/`min`/`sum` carry the usual `not-callable` (and one `assignment-from-no-return`) block disables, and the 5-column summary `select` a `call-overload` `type: ignore` (mixed aggregate funcs defeat select's typed overloads)

Both endpoints resolve through a shared `_resolve_reference_image(..., thumb=)` helper (entry lookup + traversal guard in one place, so `pylint`'s `duplicate-code` stays quiet). Reads the manifest from `get_reference_dir()` via a per-dir memoized loader; a missing/invalid `manifest.json` is treated as "no references" so every lookup 404s cleanly instead of crashing. The served image `url` points back at the images endpoint — the on-disk `path`/upstream `source_url` are never exposed. Powers the gallery/dashboard comparison panel
- `birdscanner/api/routers/settings.py` — proxies the **Settings page** to the detector's control server (prefix `/api/settings`): `GET /api/settings` → detector `/settings` (current values + restart metadata), `POST /api/settings` → detector `/settings` (partial update; relays the detector's 400 with the validation message extracted from its `{"error": …}` JSON body via `_error_detail`), `POST /api/settings/restart` → detector `/restart`. - `birdscanner/api/routers/camera.py` — proxies the detector's snapshot + crop-control endpoints (`${DETECTOR_URL}`, default `http://detector:8000`) via httpx, since the API can't open the camera itself (the detector owns it exclusively + the data volume is read-only): `GET /api/camera/snapshot` → `/capture` (cropped feed), `GET /api/camera/snapshot/full` → `/capture/full` (full-sensor preview for the crop editor), `GET /api/camera/crop` → `/crop`, `POST /api/camera/crop` → `/crop` (relays the detector's status, e.g. 400 for a bad box). Returns 503 when the detector is unreachable

**`birdscanner/db/`** — SQLite persistence layer (Phase 1):
- `birdscanner/db/models.py` — `DetectionRecord` SQLModel ORM model (`detections` table). Stores **two** confidences: `confidence` is the species-classification score (ConvNeXt), and the nullable `detection_confidence` is the object-detection score (YOLO11n on the IMX500) for the bird box (null for legacy rows written before it was persisted). Also includes nullable `box_x/box_y/box_w/box_h` floats — the detection box in normalized `[0, 1]` image fractions (null for legacy rows written before boxes were persisted) — and a nullable `video_path` (the saved mp4 clip, relative to `IMAGE_DIR`; null for legacy rows, when video is disabled, or when a single-flight-declined trigger produced no clip). When `video_path` is null the nullable `no_video_reason` string says **why** the clip is missing — `"recorder_busy"` (the single-flight recorder was busy with another sighting) or `"disabled"` (video recording is off); null when a clip exists or for legacy rows. The UI surfaces this as the tooltip on the disabled Video toggle. Three nullable **geomodel** columns record the Bayesian update when it ran: `classifier_species` / `classifier_confidence` (the classifier's own pick *before* the location prior was applied — `species`/`confidence` hold the corrected posterior) and `geo_scores` (JSON of the top pre-normalised `[species, score]` pairs, `p(y|x)·p(y|c)` before renormalising, kept for debug/display); all three are NULL when no prior was applied (no location configured / prior unavailable) and for legacy rows. A separate nullable `classifier_scores` column stores the **classifier's own top-k `[species, probability]` softmax pairs** (the raw distribution *before* any geomodel reweighting) as JSON — populated whenever the classifier could produce a full distribution (independent of the geomodel, so it is present even with no location), NULL only for legacy rows / test fakes without `predict_proba`. It powers the Lightbox **Advanced stats** panel (`frontend/src/components/AdvancedStats.tsx`). Two more nullable columns track **user species corrections** (see `birdscanner/db/corrector.py`): `corrected` (bool) is set `True` when a user overrides the assigned species from the Lightbox, and `original_species` preserves the species the system had assigned before the correction so the disagreement stays on record for retraining (the row's `confidence` still describes that prior guess; both are null for un-corrected rows). The module also defines the geomodel-prior tables: `GeoPrior` (`geo_priors` table — one row per `(species, week, probability)`, i.e. the `~706 species × 48 weeks` occurrence prior projected onto the classifier's classes) and `GeoPriorMeta` (`geo_prior_meta` — a single row `id=1` recording the `latitude`/`longitude`/`generated_at` the priors were last built for), plus `CustomSpecies` (`custom_species` table — one row per **user-added species label** not in the classifier's vocabulary, keyed by the exact-cased `name` with a `created_at`; see `birdscanner/db/custom_species.py`). All are created by `init_db`'s `create_all` (new tables need no ALTER migration)
- `birdscanner/db/database.py` — `make_engine()` / `init_db()` / `make_session_factory()`; DB path from `DB_PATH` env var. `make_engine(read_only=True)` opens the SQLite file via the `mode=ro` URI so it can be read off a read-only mount without attempting to create a journal file (used by the API). `init_db()` also runs `_migrate_detections_columns()` — a lightweight `ALTER TABLE ... ADD COLUMN` pass that backfills nullable columns added after the initial schema (e.g. `detection_confidence`, `video_path`, `no_video_reason`, `box_*`, `classifier_species`/`classifier_confidence`/`geo_scores`/`classifier_scores`, `corrected`, `original_species`), since SQLModel's `create_all` never alters an existing table. Driven by the `_DETECTIONS_ADDED_COLUMNS` map; add new post-launch columns there
- `birdscanner/db/writer.py` — `DetectionWriter`: fire-and-forget background-thread writer; `write(record)` takes a **pre-built `DetectionRecord`** (the caller names each column via the model's keyword-only constructor) and enqueues it; `stop()` flushes and exits. The writer no longer mirrors the column list in its own signature (that was `too-many-arguments`/`-locals`)
- `birdscanner/db/deleter.py` — `delete_detection(session_factory, image_dir, id)`: synchronously deletes a detection's DB row **and** its image + thumbnail + video files (best-effort file unlink — a missing file does not block the row delete; the nullable `video_path` is skipped when absent); returns `False` when no row with that id exists. Runs in the detector (which owns the read-write data volume); the API proxies delete requests to it. Unlike the writer it is synchronous (it runs on the control server's HTTP thread, not the camera callback)
- `birdscanner/db/corrector.py` — `correct_detection_species(session_factory, image_dir, id, new_species)`: synchronously reassigns a detection's classified species when a user disagrees with the model (from the Lightbox). It **moves** the still + thumbnail + clip into the corrected species' folder (best-effort `Path.replace`, mirroring the deleter — a missing/unmovable file is logged but never blocks the update, and the stored path is rewritten regardless), rewrites `image_path`/`thumbnail_path`/`video_path` to the new `{species}/` prefix, sets `corrected=True`, and records the previously-assigned species in `original_species` (only on the first correction, so repeat corrections keep the original). Leaves the model-breadcrumb columns (`classifier_species`/`geo_scores`/`classifier_scores`) untouched — a correction overrides the outcome, not the record of what the model did. Returns the updated row as a JSON-serialisable dict (`model_dump(mode="json")`), or `None` when the id is missing; a no-op when `new_species` equals the current species. Like the deleter it is synchronous (runs on the control server's HTTP thread) and validated upstream by the control-server handler against the **known** vocabulary (classifier classes ∪ custom species). It trusts the species string — a **brand-new** label (via `allow_new`) is registered as a custom species by the handler *before* this runs, so the folder it creates is a legitimate class
- `birdscanner/db/custom_species.py` — synchronous store for **user-added ("custom") species labels** (like `deleter.py`/`corrector.py`, detector-owned; no camera/ml imports). `list_custom_species(session_factory)` returns all stored labels; `add_custom_species(session_factory, name)` trims + **case-insensitively de-duplicates** (a case-variant reuses the first-stored canonical label, adding no row — idempotent) and returns the canonical name (raising `ValueError` on an empty name). Wired into the control server so a re-identification to a species outside the classifier's vocabulary can create the label and have it reappear in the picker (served via the union `GET /labels`). A custom-labelled detection is an ordinary `corrected` `DetectionRecord`; this table only records the *label* for reuse
- `birdscanner/db/geo_prior_store.py` — synchronous persistence for the geomodel prior (like `deleter.py`, detector-owned; no camera/ml imports). `read_meta(session_factory)` returns the single `GeoPriorMeta` row (or `None`); `location_matches(meta, lat, lon, tol=1e-4)` compares a stored location within a per-axis tolerance (~11 m) so float noise doesn't force a rebuild; `has_geo_priors(session_factory)` reports whether any rows exist; `replace_geo_priors(session_factory, lat, lon, priors)` clears the `geo_priors` table, bulk-inserts one row per `(species, week)` (weeks numbered `1..len`), and upserts the meta row — all in one transaction. Fed by `project_to_classifier` (`{species: [48 floats]}`). `load_geo_priors(session_factory)` is the **inverse read** (`{species: [prob_week_1..N ordered by week]}`, `{}` when empty) used by `detector/pipeline/gating.build_geo_adjuster` at startup to build the in-memory runtime `GeoPriorAdjuster`

**`tools/build_species_reference.py`** — offline species-reference data-bank builder (**not** part of the runtime; run on a dev machine with internet):
- Iterates every label in `assets/models/convnext_v2_tiny.onnx_class_to_idx.json` (the source of truth for "what species exist"; the `"Unknown"` sentinel is skipped) and fetches, per species: the **Wikipedia** REST summary (`summary` + lead image + canonical URL + Wikidata id), a best-effort "Behaviour"/"Habitat" section (`behaviour`) via the MediaWiki API, image attribution/license via `imageinfo`+`extmetadata`, and the **Wikidata** taxon name (P225) for `scientific_name`. `urllib` for the network; a **`tqdm`** progress bar (dev-only dep, `requirements.dev.txt`) tracks the species being processed; descriptive `User-Agent`, polite throttling
- Writes `assets/species_reference/manifest.json` (schema: `{version, generated_at, source, species:{<label>:{common_name, scientific_name|null, summary, behaviour|null, wikipedia_url|null, images:[{path, thumbnail_path?, source_url, attribution, license|null}]}}}`) + `coverage_report.json` (buckets: `missing`, `no_images`, `no_scientific_name`, `skipped`), and downloads each reference image to `assets/species_reference/images/<slug>/<n>.jpg` (`<slug>` = label lowercased, non-alphanumeric runs → single `-`) **plus a small `<n>_thumb.jpg` thumbnail** beside it (128px centre-cropped square, `thumbnail_path` in the manifest — served to tiny displays like the "In season" panel so they never pull the multi-MB original). The pure, offline thumbnail helpers live in **`tools/reference_thumbnails.py`** (split out to keep the builder module under pylint's `too-many-lines`; imports only `os` + Pillow, no network): `make_thumbnail` (Pillow, `Image.Resampling.LANCZOS`) does the resize, and `ensure_thumbnails(species_map, output_dir)` runs at the end of a build as an idempotent **backfill** that generates any missing thumbnails from the on-disk originals, so an already-built bank gains thumbnails on a plain re-run. The per-species image download + thumbnail is factored into `_download_lead_image` (keeps `build_species_entry` under `too-many-locals`)
- **Incremental / idempotent**: re-runs only fetch labels missing from the manifest (or whose image files are gone), so adding a new class = add the label to the classifier's `class_to_idx.json` + re-run. When only an image file is missing but the cached text (`summary`/`scientific_name`/`behaviour`) is intact, that species takes a fast **image-only** path (`can_refresh_images_only` / `refresh_species_images`): the thumbnail is re-downloaded from the manifest's stored `source_url` reusing the cached attribution/license — **one** request, **zero** Wikipedia/Wikidata text calls (vs. the ~6 sequential round-trips of a full rebuild, which is what made a "just fetch the missing thumbnails" re-run crawl on the Pi). `build_manifest` first classifies work into a `(label, mode)` todo list via `_plan_todo` (`"image"` vs `"full"`), then iterates it under the tqdm bar. Flags: `--force` (refetch all, forces every label to `"full"`), `--limit N` (only the first N labels needing a fetch — for a quick live smoke test), `--throttle SECONDS`; these are bundled into a `BuildOptions` dataclass passed to `build_manifest(labels, overrides, existing, options)`. A fourth flag **`--output-dir DIR`** (default `$SPECIES_REFERENCE_DIR`, else the in-repo `assets/species_reference/`) chooses **which bank** to build/backfill into: `main` calls `_set_output_dir(args.output_dir)` to repoint the artifact-path module globals (`OUTPUT_DIR`/`MANIFEST_PATH`/`COVERAGE_PATH`), so you can run the thumbnail backfill **directly against the deployed data-volume bank** (e.g. `--output-dir /mnt/ssd/birdfinder-data/species_reference`) instead of building the repo bank and copying it over. `OVERRIDES_PATH` is deliberately **not** repointed — `overrides.json` is curated source and is always read from the repo regardless of `--output-dir`
- `assets/species_reference/overrides.json` — hand-maintained `{<label>:{wikipedia_title?, scientific_name?, skip?}}`; consulted first, so it fixes name→article mismatches (e.g. apostrophe-stripped labels like `Audouins gull` → `Audouin's gull`) and forces `skip` for `"Unknown"`. This file is **source** (curate + commit it); the generated `manifest.json`/`coverage_report.json` and the downloaded `images/` are **build artifacts** shipped/mounted out-of-band (like the ONNX model — a full build is ~hundreds of MB of JPEGs), and are `.gitignore`d (a `!assets/species_reference/overrides.json` negation keeps the curated overrides tracked despite the repo-wide `*.json` rule). The API finds the data via `SPECIES_REFERENCE_DIR` (default `assets/species_reference/`). **In the Docker stack** the bank is **not** baked into the api image (`Dockerfile.api` copies only `birdscanner/` + `frontend/dist`); instead it lives on the data volume at `/data/species_reference` with `SPECIES_REFERENCE_DIR=/data/species_reference` (set in `.env.example`). Build it on a dev machine and copy `manifest.json` + `images/` into `<data-volume>/species_reference/` on the host (e.g. `/mnt/ssd/birdfinder-data/species_reference/`); the api mounts the data volume read-only and reads it there. If this dir is missing/empty, **every** reference lookup 404s and the Lightbox panel shows "No reference available" for all species — that is the symptom of an un-deployed bank, not a code bug. **Gotcha (thumbnails):** the builder writes to the **in-repo** bank by default, so building/backfilling there and forgetting to redeploy leaves the api serving a *different* (thumbnail-less) bank on the data volume — the reference `/thumbnail` endpoint then silently falls back to the full-res original (the "In season" icons stay multi-MB). Either copy the updated `manifest.json` + `images/` (now with `*_thumb.jpg` siblings) onto the data volume **and restart the api** (its manifest is memoized per-directory, cleared only by a process restart), or run the backfill straight against the deployed bank with `--output-dir /mnt/ssd/birdfinder-data/species_reference` (see the `--output-dir` note above) and restart the api

**`tools/build_geomodel_map.py`** — offline builder for the geomodel↔classifier label crosswalk (**not** runtime; pure stdlib, no network). Reads the geomodel labels (`assets/labels/BirdNET+_Geomodel_V3.0.3_Global_12K_Labels.txt`) + the classifier's `assets/models/convnext_v2_tiny.onnx_class_to_idx.json`, calls `geomodel.build_name_mapping`, and writes `assets/labels/geomodel_classifier_map.json` (the sorted **`{classifier_label: geomodel_common_name}`** map). It **prints the unmatched classifier classes as a table** for a human to curate. Curated entries are added straight into the JSON as more `"<classifier_label>": "<geomodel_common_name>"` pairs and are **preserved** across re-runs — `load_existing_overrides` treats any existing entry that differs from what auto-matching alone would produce (a class that doesn't auto-match, or one corrected to a different geomodel species) as a curated override and feeds it back in. `--check` prints the table without writing. Run with the project `.venv` (it imports `birdscanner.ml.geomodel`, which imports `onnxruntime`)

**`birdscanner/ml/classification.py`** — ONNX classifier stack:
- `ONNXClassifier` — raw ONNX Runtime wrapper; expects `(N, C, H, W)` float32
- `Classifier` — adds preprocessing and class-index mapping; `classify()` returns `(species_str, confidence_float)` (argmax); `predict_proba()` returns the full `(num_classes,)` softmax vector for callers that need the whole distribution (the geomodel Bayesian update)
- `build_preprocessing` — pure PIL+NumPy pipeline (resize → center crop → ImageNet normalize → NCHW)
- `top_k_predictions(probs, idx_to_class, k)` — pure helper returning the `k` highest `(label, probability)` pairs of a softmax vector (mirrors `GeoPriorAdjuster._top_scores`); used by `classification_pipeline._predict_species` to persist the classifier's top-k distribution (`classifier_scores`) for the Advanced-stats panel

**`birdscanner/detector/config/settings.py`** — user-editable runtime settings (pure domain, unit-tested in `tests/detector/test_settings.py`):
- Layers a JSON overlay on top of the static `config` defaults so the **Settings page** can change detection parameters without editing code or rebuilding. `Settings` is the dataclass of editable values: `detection_threshold`, `classification_threshold` (the save-side floor), `ignore_species`, `excluded_classes` (the object-detection/YOLO-class exclude list — labels like `"bench"` dropped before tracking so false positives don't flood the logs; validated as a trimmed/de-duped string list by the shared `_as_string_list`, matched case-insensitively; live), `stability_seconds`, `image_dir`, `video_save`/`video_pre_roll_seconds`/`video_post_roll_seconds`, `multithread`, `debug`, `latitude`/`longitude` (the deployment location driving the geomodel prior — `Optional[float]`, **default `None` = unset**; a set value is range-validated lat `[-90, 90]` / lon `[-180, 180]`, `None` clears it — `_as_optional_ranged_float` handles the null-allowed case). `excluded_classes` defaults to `{"bench"}` (seeded from `config.excluded_classes`)
- `LIVE_FIELDS` / `RESTART_FIELDS` partition the model: **live** fields (`detection_threshold`, `classification_threshold`, `ignore_species`, `excluded_classes`, `debug`) are read by a running detector and take effect immediately; the rest are consumed once at startup wiring, so a change is persisted but only takes effect on the **next detector boot** (`RESTART_FIELDS` drives the UI's "restart" badge — `latitude`/`longitude` are restart fields because the prior is only rebuilt at boot). `default_settings()` seeds from the current `config`/`IMAGE_DIR` so an existing deployment is unchanged until the user edits something
- `merge_settings(current, updates)` validates + coerces a partial update (unknown key / out-of-range value → `ValueError` **without** mutating state, so the caller returns 400); `load_settings`/`save_settings` (atomic) persist at `settings_config_path()` (env `SETTINGS_PATH`, default `/data/settings.json`, alongside `crop.json`). Any read error falls back to defaults so a corrupt file never blocks startup

**`birdscanner/detector/config/settings_controller.py`** — applies settings to the running detector (mirrors `crop_controller.py`; unit-tested in `tests/detector/test_settings_controller.py`):
- `apply_settings_to_config(settings)` runs **once at startup** (before the pipeline is wired) so restart-only settings shape the initial build; it pushes every field onto the `config` singleton + `classification_pipeline.IMAGE_DIR`. `apply_settings_to_context(settings, context)` applies the save-side fields (`save_confidence_threshold`, lower-cased `ignore_species`) to the `PipelineContext` — these live on the context (not `config`) because they are consumed inside `ml/`, which must not import `detector/`
- `SettingsController(path, boot_settings, context)` handles **runtime** updates: `update(updates)` validates + persists + live-applies (mutates `config.threshold`, `config.excluded_classes` (lower-cased), the context's save-fields, and the `tracking` log level) and returns `get_state()` (`{settings, needs_restart, restart_fields, live_fields}`); `needs_restart` is `True` when a restart-only field differs from the booted value, so the UI can prompt for a restart

**`birdscanner/detector/config/config.py`** — application configuration:
- `Config` dataclass + module-level `config` instance; edit values here instead of passing flags. To keep the top-level dataclass small (`too-many-instance-attributes`), the settings are grouped into two nested sub-configs: **`config.intrinsics`** (`IntrinsicsConfig`: the network-intrinsics overrides `main.py` pushes onto the IMX500 intrinsics object — `fps`, `bbox_normalization`, `bbox_order`, `ignore_dash_labels`, `preserve_aspect_ratio`, `labels`) and **`config.video`** (`VideoConfig`: `save` default `True`, `pre_roll_seconds` default `3.0`, `post_roll_seconds` default `4.0`, `full_fov` default `False` — when `False` the clip records the cropped ISP `main` frame (matching the still); `True` records the whole uncropped FOV from the raw stream, which is softer/noisier, 4:3, and costs extra CMA). The remaining top-level fields are `model`, `threshold`, `print_intrinsics`, `multithread`, `object_duration_threshold`, `excluded_classes` (`set[str]`, default `{"bench"}` — the object-detection/YOLO-class exclude list read live in the capture loop; see `filter_excluded_detections`), `debug`, `preview`, and `latitude`/`longitude` (`Optional[float]`, default `None` — the geomodel-prior location). `main.py` imports the instance as `app_config` (to avoid clashing with the local `config` returned by `picam2.create_preview_configuration`) and its intrinsics-override loop iterates `vars(app_config.intrinsics)`. The best-frame selector is always built by `build_gating`; the video recorder is wired only when `save_video` is set

**`birdscanner/detector/paths.py`** — package-anchored data-file resolution:
- Resolves the detector's on-disk data files relative to the **package location** (`<repo>/assets`, `/app/assets` in Docker), not the current working directory — this is what lets the detector run as `python -m birdscanner.detector.main` from anywhere (the old `cwd=src` requirement is gone). `assets_dir()` / `model_dir()` are overridable via the `ASSETS_DIR` / `MODEL_DIR` env vars (default `assets/` and `assets/models/`). Helpers `coco_labels_path()`, `class_to_idx_path()`, `classifier_model_path()`, and the geomodel trio `geomodel_model_path()` / `geomodel_labels_path()` / `geomodel_map_path()` (the ONNX under `MODEL_DIR`, the `*_Labels.txt` + `geomodel_classifier_map.json` under `assets/labels/`) return the concrete files; `main.py` uses them instead of hardcoded relative strings

**`birdscanner/detector/main.py`** — entry point + capture loop only:
- `main()` reads as a short startup script (`load_settings`/`apply_settings_to_config` → `configure_logging` → `make_engine`/`init_db` → `refresh_geo_priors` → `wait_for_camera`/`prepare_intrinsics` → `setup_classifier` → `build_gating` → `DetectionWriter`/`build_geo_adjuster`/`build_manager`/`apply_settings_to_context`/`SettingsController` → `build_camera` → maybe wire `video_frame_source` → `_start_control_server` → `_run_capture_loop`). Right after `build_camera` it sets `manager.context.video_frame_source = build_clip_frame_source(camera.picam2)` **only when `config.video.full_fov` is on** (and video is enabled), so the clip records the full uncropped FOV from the raw stream (see `detector/raw_frame.py`). In the default cropped-clip mode this is left unwired, so the recorder receives the cropped `main` frames and the clip matches the still. Right after `init_db` it calls `refresh_geo_priors(make_session_factory(engine), app_config.latitude, app_config.longitude)` (a no-op unless the configured location changed) so the geomodel prior is current before the pipeline runs. The persisted **settings overlay** is loaded and pushed onto `config` **before** anything is wired so restart-only settings shape the build; a `SettingsController` is then handed to `_start_control_server`. `_schedule_restart` (the `/restart` callback) does a delayed `os._exit(0)` from a timer thread so the HTTP response flushes first and Docker's `restart: unless-stopped` policy relaunches the detector to pick up restart-only settings (outside Docker it just exits). The camera bring-up lives in `detector/hardware/camera.py` and the classification-pipeline wiring in `detector/pipeline/gating.py`; `main.py` keeps only `_start_control_server`, `_run_capture_loop`, `_shutdown`, and `main`. A `KeyboardInterrupt` out of `_run_capture_loop` triggers `_shutdown`
- Reads all runtime settings from `config.config` (imported as `app_config`); no `argparse`
- `_run_capture_loop` installs the per-frame `detection_callback` (updates the tracker + queues bird detections via `process_detections`) then loops: it holds the `CropController`'s `camera_lock` across `capture_metadata()` + `parse_detections()` so a UI-triggered reconfigure (which may `stop()`/`start()` the camera) never races an in-flight capture. It then passes the parsed detections through `filter_excluded_detections(results, labels, app_config.excluded_classes)` before storing them, so excluded object-detection classes (e.g. `"bench"`) never reach the tracker or drawing — this is the single point that keeps false-positive classes out of the logs; the exclude set is read live from config so the Settings page can edit it without a restart
- Creates the SQLite engine and runs `init_db()` on startup **before camera init**, so the schema always exists; the detector owns all DB writes, so this lets the read-only API serve an empty gallery even when the camera never comes up. The same `engine` is reused to wire a `DetectionWriter` into the `ClassificationManager` (via `build_manager`) so every high-confidence classification is persisted, and to bind the delete **and species-correction** callbacks for `_start_control_server` (which also receives the classifier vocabulary — `sorted(classifier.idx_to_class.values())` minus the `Unknown` sentinel — to serve `GET /labels` and validate corrections); the writer is flushed via `detection_writer.stop()` on `KeyboardInterrupt`
- `_start_control_server` wires the `camera_server` (snapshots + crop + detection deletes) to the started camera and the DB engine; a port-bind `OSError` is logged and swallowed so the detection pipeline keeps running without it

**`birdscanner/detector/hardware/camera.py`** — camera bring-up (IMX500 + Picamera2 + crop):
- `wait_for_camera()` wraps `IMX500(...)` in a retry-with-backoff loop (30 s default): when the camera dev-node is missing it logs a concise warning and retries instead of crashing, so the detector never crash-loops and auto-recovers when the camera reappears
- `prepare_intrinsics(imx500)` validates the object-detection task and iterates `vars(app_config.intrinsics)`, so optional intrinsic fields left as `None` do not clobber the network intrinsics defaults (and `print_intrinsics` prints + exits)
- `build_camera(imx500, intrinsics)` loads the persisted crop region via `load_crop_region(crop_config_path(), default_crop_region(...))` (default = the historical 900×900 at `(4/13, 5/10)` of the 4056×3040 sensor, aimed at the feeder), sizes the `main` stream to the crop's aspect ratio (`main_stream_size_for_crop`), applies `vflip=True, hflip=True` (camera is mounted upside-down), starts the camera, and builds the `CropController`. The camera config is built through a `build_camera_config(main_size, scaler_crop)` closure that centralises every picamera2 knob so `CropController` can rebuild it on a reconfigure. **Only when `config.video.full_fov` is enabled** does it request the full-FOV **`raw`** stream (via `_full_fov_raw_stream(picam2)`, which picks the 2028×1520 mode's **unpacked** format so `make_array("raw")` yields a directly-demosaicable array) — the uncropped source for the full-FOV video clip (`raw_frame.py`). In the default cropped-clip mode `raw_stream` is `None` and the `raw=` key is omitted from the config, so the explicit unpacked-raw request (and its ~+14 MB CMA premium across the 6 buffers vs. packed) is not paid — freeing headroom that offsets the `DEFAULT_LONG_SIDE` 640→1280 bump. The raw stream is already allocated regardless, so requesting it explicitly only makes it accessible in the callback. `_full_fov_raw_stream` **must not** read `picam2.sensor_modes`: that property lazily calls `configure()` (allocating full-res DMA buffers) for **every** sensor mode, which exhausts the container's host-global CMA pool and crashes bring-up with `OSError: [Errno 12] Cannot allocate memory` **before** `picam2.start()` (this was the PR #81 regression). Instead it enumerates raw formats via the **non-allocating** `picam2.camera.generate_configuration([libcamera.StreamRole.Raw])` (the cheap part of `sensor_modes`) and derives the unpacked format with `SensorFormat(name).unpacked`; passing an explicit size **and** format means `configure()`/`start()` never re-consults `sensor_modes` either. On any failure it falls back to a size-only spec (clips degrade to the cropped `main` frame, never crash). Returns a `Camera` bundle (picam2 + imx500 + intrinsics + crop_controller). This module is Pi-only (imports `libcamera`/`picamera2`); no test imports it

**`dev/emulation/` + `dev/run_emulated.py`** — the **off-Pi camera emulator** (dev/test-only; imports **no** picamera2/libcamera — it *is* the stand-in). It lets the **real** `camera.py` bring-up and the **real** `main._run_capture_loop` execute off the Pi, which is what finally gives that live path test coverage (previously the only 0%-coverage code) and enables local development without the hardware. Mechanism: install fake `libcamera`/`picamera2` **modules** into `sys.modules` **before** `birdscanner.detector.hardware.camera` is imported (it imports them at module load), so the real code runs unchanged against the fakes. `camera.py`/`main.py` are **not modified**.
- `emulation/yolo.py` — `OnnxYoloDetector`: the real off-Pi object detector (there is no off-Pi `.rpk`). Runs a stock YOLO11n `.onnx` via the already-pinned `onnxruntime`, decoding the `(1, 84, 8400)` output (letterbox → NMS) into `Detected(box_xyxy_norm, score, coco_name)`. A `Detector` Protocol lets tests inject a deterministic stub instead of the model. **Label-alignment gotcha:** it emits the class *name*, not the raw YOLO index — the fake IMX500 re-indexes by name against the intrinsics label list, because the `.rpk` and the generic export don't share a label ordering (a raw index would file a detected bird under the wrong class and it would never gate as `"bird"`). `LetterboxTransform` is the value object bundling the scale/pad + its `to_normalized(box)` inverse.
- `emulation/frames.py` — `FrameSource` Protocol + `TestImagesSource` (cycles the bundled `tests/_test_images` JPEGs — zero setup, deterministic) and `VideoSource` (reads an mp4/mov via `cv2`, BGR→RGB, optional loop). Both yield `(H, W, 3)` uint8 RGB.
- `emulation/fakes.py` — the module fakes mirroring exactly the surface the real code uses: `FakeIMX500` (`get_outputs` emits the `.rpk` tensor convention — xy-ordered xyxy scaled to `input_h`, classes name-mapped to the filtered intrinsics labels — so the real `parse_detections`/`_decode_boxes` run faithfully; `convert_inference_coords` scales the decoded box to the fake `main` size), `FakeNetworkIntrinsics`, `FakePicam2` (its `capture_metadata` is the frame pump: pull next frame → resize to `main` size → run detector → fire `pre_callback` → return metadata; a `max_frames` cap raises `KeyboardInterrupt` so the real loop unwinds via `main`'s existing shutdown), `FakeRequest`, `FakeMappedArray` (writable `.array`), plus `FakeTransform`/`FakeSensorFormat`. A module-level `_EmulationState` (frame source + detector + the shared `FakeIMX500`) is read by the fakes, since `build_camera` constructs `Picamera2(camera_num)` with no reference to the IMX500.
- `emulation/install.py` — `install_fake_camera_modules(frame_source, detector, max_frames=None)` registers the fakes in `sys.modules` and installs the state; `uninstall_fake_camera_modules()` reverses it (tests clean up via the `camera_emulator` fixture in `tests/detector/conftest.py`).
- `run_emulated.py` — the dev entry point: `python -m dev.run_emulated`. Installs the fakes from env (`FAKE_CAMERA_SOURCE` = a video path, else the bundled test images; `YOLO_ONNX_PATH` default `assets/models/yolo11n.onnx`; `FAKE_CAMERA_MAX_FRAMES` optional cap) **before** importing `main`, then runs the real `main()` — so the ConvNeXt classifier, geomodel prior, SQLite writer, control server, and clip encoding all run for real. Point the API at the same `DB_PATH`/`IMAGE_DIR` to watch the emulated detections appear in the UI. Needs `assets/models/yolo11n.onnx` (out-of-band, git-ignored like the classifier ONNX — export with `yolo export model=yolo11n.pt format=onnx`; see `requirements.dev.txt`); absent, the emulator + its tests skip/degrade gracefully.

**`birdscanner/detector/pipeline/gating.py`** — classification-pipeline wiring:
- `build_gating(intrinsics)` builds the stable-track gating machinery and returns a `Gating` bundle (tracker, best-frame selector, video recorder). It constructs the `StableDetectionTracker` (wired to a `TrackingLogger` for lifecycle logging), a `BestFrameSelector` freed per-track via the tracker's combined `on_track_deleted` callback (a non-optional local captured by that closure so it never dereferences a possibly-`None` selector), and — when `save_video` is set — a `VideoRecorder`. `min_stable_frames(fps)` converts `object_duration_threshold` × fps into a frame count (floored at 1, so `object_duration_threshold <= 0` classifies after a single stable frame)
- `build_manager(classifier, gating, detection_writer, geo_adjuster=None)` assembles the `PipelineContext` and `ClassificationManager` from the `Gating` bundle; the recorder's `add_frame`/`trigger` are injected as `video_frame_fn`/`record_fn` (keeping picamera2 out of `ml/`), and the optional `geo_adjuster` is set on the context. It builds the async manager with an explicit `queue_maxsize=CLASSIFICATION_QUEUE_MAXSIZE` (module constant, **32**) so the classification backlog is bounded and excess frames are dropped rather than accumulated — see the `ClassificationManager` note above for why an unbounded queue leaked memory and stalled the camera. Imports `ml/`, `config`, `track_logging`, `video_recorder`, `db/writer`, and `db/geo_prior_store` — no picamera2, so it is importable without a camera
- `build_geo_adjuster(classifier, session_factory)` builds the runtime `GeoPriorAdjuster` from the DB-stored prior (`load_geo_priors`) + the classifier's `idx_to_class`; returns `None` (geomodel correction **off**, classification unchanged) when the classifier has no class map or no priors are stored (no location configured, or the startup rebuild was skipped/failed). `main.py` calls it after `refresh_geo_priors` and passes the result to `build_manager`

**`birdscanner/detector/pipeline/geo_priors.py`** — startup orchestration for the geomodel prior (unit-tested in `tests/detector/test_geo_priors.py`):
- `refresh_geo_priors(session_factory, lat, lon, *, compute=...)` runs once at startup (after `init_db`). It **skips** when no location is set (`lat`/`lon` is `None`) or when the stored `GeoPriorMeta` already matches the configured location **and** rows exist (`location_matches` + `has_geo_priors` — a cheap no-op on a normal restart); otherwise it computes the `{species: [48 floats]}` prior and calls `replace_geo_priors`. It **never crashes startup**: a missing geomodel model file or an inference error is logged and swallowed (mirroring `wait_for_camera` and the classifier-absent skips), and an empty result stores nothing. `compute` defaults to the real `compute_classifier_priors` (wired through the `geomodel_*_path()` helpers) but is **injectable** so tests avoid the 28 MB ONNX / onnxruntime. Ties together `ml/geomodel` (compute) + `db/geo_prior_store` (persist); imports both, like `gating.py`, so it stays testable off-Pi

**`birdscanner/detector/hardware/video_recorder.py`** — on-demand short-clip recorder (unit-tested in `tests/test_video_recorder.py`):
- `VideoRecorder` keeps a bounded `deque` of recent RGB frames (the pre-roll) fed by `add_frame` every frame — **cheap, no encoding while idle**, which matters because the **Pi 5 has no hardware video encoder** so all encoding is software (CPU). By default the frames it is fed are the **cropped, ISP-processed `main` frames** — the same source as the saved still — so the clip matches the still (same crop, aspect, ISP quality) at the `main` stream resolution (`DEFAULT_LONG_SIDE=1280` on the long edge). When `config.video.full_fov` is on it is instead fed the **full uncropped FOV** (debayered from the raw stream by `raw_frame.py`). The output size is auto-sized from `frames[0].shape` at encode time. `trigger(dest_path)` snapshots the pre-roll, keeps collecting `post_roll_seconds` of live frames, then encodes the whole sequence to an mp4 (`cv2.VideoWriter`, RGB→BGR like the still writes) on a **background thread** so the camera callback never blocks. The clip is encoded as **H.264 (`avc1`)** — the only codec browser `<video>` players can decode — via `_open_writer`, which tries the codecs in `_CODEC_PREFERENCE` (`avc1`, then `mp4v`) and uses the first whose writer opens; `mp4v` is a last-resort fallback for an FFmpeg build without an H.264 encoder (logged as a warning, since an mp4v clip will **not** play in the browser — it renders as the poster/thumbnail, which looks like a single frozen frame). Encode quality is raised by setting `OPENCV_FFMPEG_WRITER_OPTIONS=crf;20` at module import (via `os.environ.setdefault`, so an operator can override; a build whose FFmpeg ignores it just uses its default bitrate) — OpenCV's default H.264 bitrate otherwise looks heavily compressed/blocky. **Single-flight**: a trigger while a clip is recording is declined (returns `False`), bounding CPU/RAM. Durations come from `config` (`video_pre_roll_seconds`/`video_post_roll_seconds`); fps from the camera inference rate. Trade-off: the RAM buffer costs ~1 MB/frame × pre-roll frames at 640, ~4× that at 1280 — downscale/limit if it bites

**`birdscanner/detector/hardware/raw_frame.py`** — full-FOV clip frames from the raw sensor stream (unit-tested in `tests/detector/test_raw_frame.py`; pure `cv2`/`numpy`, no picamera2, so it runs off-Pi). **Only used when `config.video.full_fov` is enabled** (opt-in); the default cropped-clip mode never touches this module:
- The stills/preview/`main` stream are cropped to the `ScalerCrop` detection region; the full-FOV clip is meant to show the **whole uncropped scene**. There is only **one `ScalerCrop` per camera** and it applies to every *processed* stream (`main`/`lores`), so the only source of the full FOV is the **raw** stream (the always-allocated 2028×1520 binned readout, unaffected by `ScalerCrop`). This module turns a raw Bayer frame into a downscaled RGB frame for the recorder
- `bayer_cv2_code(raw_format)` — maps a libcamera raw format string (`SBGGR`/`SGBRG`/`SGRBG`/`SRGGB…`) to the OpenCV demosaic code. The letters are **swapped vs. libcamera** (OpenCV names the 2×2 tile one pixel in), so `SBGGR→COLOR_BAYER_RG2RGB`, `SRGGB→COLOR_BAYER_BG2RGB`, etc.
- `RawToRgb(raw_format, long_side=640)` / `.convert(raw)` — right-shifts samples to 8-bit (by the format's bit depth), `cv2.cvtColor` demosaics to RGB, then `cv2.resize(INTER_AREA)` downscales so the longer edge is `long_side`, preserving aspect (≈640×480 for the 4:3 FOV)
- `build_clip_frame_source(picam2)` — reads the **actual configured raw format back** from `picam2.camera_configuration()` (so the Bayer pattern matches the sensor's current vflip+hflip state) and returns a `request → RGB frame` callable wired into `PipelineContext.video_frame_source` (in `main.py`, only when video is enabled **and** `config.video.full_fov` is on). The callable never raises: any failure logs and returns `None`, so the pipeline degrades to the cropped `main` frame. NB: the exact Bayer order under the flip + the RGB channel order are hardware-specific — if clip colours ever invert or the demosaic looks wrong, check the format→`cv2` code mapping here first (mirrors the RGB/BGR gotcha for the stills)

When the detection is `corrected`, the `% match` reading is replaced by `✎ Corrected by you` (Fraunces italic, `gold-deep`) with a `bark` footnote `model saw {confidence}% {original_species}`. `Timeline`/`Gallery`/`History`/`Dashboard` pass **both** `onDelete` and `onUpdate` through — the update callback replaces the detection in place in each page's local list(s) (Dashboard maps `today`+`recent`, History maps `detections`), mirroring the `onDelete` chain. **Synchronised image swap on prev/next**: the lightbox holds a `shown` detection in state *decoupled from the incoming `detection` prop* and derives **everything** (image, box overlay, caption, reference species, **advanced stats**) from `shown`. On navigation it does **not** swap immediately — it keeps the current plate fully rendered and **preloads the next full image off-screen** (`new Image()`), only committing `setShown(detection)` once the bytes have decoded, so the picture, its box and its label all arrive **together** (previously the box/caption jumped to the next sighting a beat before its network-fetched image caught up — the reported clunk). A same-id update (in-place species correction) applies immediately with no reload, since the bytes are unchanged. The swapped-in image is keyed on the shown `id` so it remounts on each navigation and **quietly resolves from a soft `blur(4px)` into focus** (`animate-plate-develop`, a gentle 0.4s cross-fade with no scrim or moving highlight) — deliberately low-key. The keyframe lives in `index.css` and is zeroed by the global `prefers-reduced-motion` guard. Keeps Esc/←/→ keyboard nav, prev/next arrows, body-scroll lock, and backdrop-click-to-close
- `frontend/src/components/FileDownloader.tsx` — manages bulk-select toolbar (select all / clear / count); streams ZIP download via `fetch()` + `ReadableStream`; tracks progress from `Content-Length` header; triggers browser download via `URL.createObjectURL`. Also renders a bulk **Delete** button: confirms, deletes the selected ids one-by-one via `api.detections.delete`, and reports the successfully-deleted ids to `onDeleteSelected` (partial failures still remove what succeeded)
- Charting: the Statistics page uses **Nivo** (`@nivo/core`/`bar`/`line`/`heatmap`); the older hand-rolled SVG charts (`NetworkMonitor`) stay as they are. Nivo is lazy-loaded (see `App.tsx`) so it lands in its own chunk

- Dev: `npm run dev` proxies `/api/*` to `http://localhost:8080`

**`birdscanner/detector/hardware/camera_server.py`** — detector control HTTP server (camera snapshots + crop control + detection deletion):
- `encode_jpeg(frame)` / `capture_jpeg(picam2)` — encode an RGB frame (RGB→BGR for OpenCV; the stream is `BGR888`, see "Colour channel order") / capture one from the `main` stream and encode it
- `start_camera_server(picam2, port, deps=None)` — starts a `_ControlServer` (a `ThreadingHTTPServer` subclass that carries `picam2` + a `ControlServerDeps` bundle) on a background daemon thread. The optional deps are collapsed into the `ControlServerDeps` **value object** (`crop_controller`, `delete_detection`, `settings_controller`, `restart`, `correct_species`, `species_labels`, `list_custom_species`, `register_species`) rather than separate arguments, so the factory/`__init__` stay under `too-many-arguments`. The request handler is a **module-level** `CameraRequestHandler` that reads those deps off `self.server.deps` — not a per-server closure (that nesting made the factory trip `too-many-statements`). Routes: `GET /capture` (fresh JPEG of the cropped feed); when a `crop_controller` is supplied, `GET /capture/full`, `GET /crop`, `POST /crop`; when a `delete_detection` callback is supplied, `DELETE /detections/{id}` (204/404/500); when a `correct_species` callback is supplied, `PATCH /detections/{id}` (species correction from a `{"species": …, "allow_new"?: bool}` body — validates the species against the **known** vocabulary via `_known_species()` = `species_labels` ∪ `list_custom_species()` (read live so runtime-added labels count), returning a **400 JSON** `{"error": …}` for an unknown one so the message survives the API proxy, 200 with the updated record dict on success, 404 when the id is missing). **New-species path**: when the label is unknown but `allow_new` is truthy and a `register_species` callback is wired, `_resolve_new_species` first reuses any case-insensitive match against a known label (so a differently-cased spelling never duplicates), else length-checks (`_MAX_SPECIES_NAME_LEN=100`) and calls `register_species` to persist a new custom species, then corrects to the canonical name — so a bird outside the classifier's vocabulary can be recorded; without `allow_new` an unknown label stays a 400. When `species_labels` or `list_custom_species` is supplied, `GET /labels` returns the **sorted union** as `{"species": [...]}`, powering the correction picker; when a `settings_controller` is supplied, `GET /settings` (current values + restart metadata) and `POST /settings` (partial update — a validation `ValueError` becomes a **400 JSON** `{"error": msg}`, not send_error's HTML, so the message survives the API proxy; an `OSError` on persist → 500 JSON); and when a `restart` callback is supplied, `POST /restart` (invokes it, returns 202). Each dependency independently enables its routes; when one is absent its routes 404. `camera_server_port()` reads `CAMERA_SERVER_PORT` (default 8000)
- Wired into `main.py` after `picam2.start(...)` so the read-only API can surface a live "Test Camera" image, the crop editor, delete detections, **and correct a detection's species** even though the detector owns the camera + read-write data volume exclusively; the delete + correction callbacks are bound to the same engine the `DetectionWriter` uses (see `birdscanner/db/deleter.py` / `birdscanner/db/corrector.py`). Shut down on `KeyboardInterrupt`. NB: because the server starts only after the camera initialises, deletes/corrections and the vocabulary (like snapshots/crop) are unavailable while the camera is down — the Lightbox picker shows an offline state, which is fine since a correction needs the detector up to persist anyway

**`birdscanner/detector/hardware/crop.py`** — pure, camera-independent crop domain (fully unit-tested in `tests/test_crop.py`):
- `CropRegion` — `(x, y, w, h)` in **unflipped raw sensor pixels**; `.clamped()` enforces `MIN_CROP_PX` and sensor bounds (`SENSOR_W=4056`, `SENSOR_H=3040`)
- `NormalizedBox` (`nx,ny,nw,nh`) and `SensorDimensions` (`w,h`) — small `NamedTuple` value objects that carry the UI box and sensor dims so the conversion functions (and `CropController`) don't take four/six scalar args
- `default_crop_region()` — the historical 900×900 feeder crop
- `normalized_to_sensor(box, sensor=SensorDimensions())` / `sensor_to_normalized(region, sensor=SensorDimensions())` — convert between a `NormalizedBox` (fractions over the displayed preview) and sensor coords via a **direct per-axis scale, no rotation**: libcamera applies `ScalerCrop` in the same orientation as the (vflip+hflip) transformed preview, so the displayed box maps straight to the matching sensor region (verified empirically — inverting the rotation cropped the diagonally-opposite corner)
- `main_stream_size_for_crop()` — the `main` ISP output size matching a crop's aspect ratio (longer edge `DEFAULT_LONG_SIDE=1280`, even-aligned), so a non-square crop is not stretched. This sizes the saved still, the classifier ROI, **and** (in the default cropped-clip mode) the video clip — the one lever for their shared resolution. It was bumped 640→1280 to sharpen the still/clip; real detail is still capped by the crop region's sensor-pixel size, and raising it grows the `main` DMA buffers (~4× vs 640), so drop it back toward 640 if the detector ENOMEMs at `picam2.start()`
- `load_crop_region()` / `save_crop_region()` (atomic) — JSON persistence at `crop_config_path()` (env `CROP_CONFIG_PATH`, default `/data/crop.json`); any read error falls back to the default so a corrupt file never blocks startup

**`birdscanner/detector/hardware/crop_controller.py`** — `CropController(picam2, config)` applies + persists the crop on the running camera, serialising all camera access behind `camera_lock` (an `RLock` shared with `main.py`'s capture loop). Its static configuration (starting `region`, `main_size`, `config_factory`, `config_path`, and `sensor` dims) is bundled into a `CropControllerConfig` dataclass so the constructor takes just the live `picam2` handle + that config:
- `set_from_normalized()` / `reset_to_default()` → `_apply()`: a **pan/zoom at the same aspect ratio** applies live via `set_controls({"ScalerCrop": ...})`; an **aspect-ratio change** triggers a `stop()`→`configure()`→`start()` reconfigure (using the `config_factory` from `main.py`) to resize the `main` stream
- `capture_full_preview_array()` — briefly widens `ScalerCrop` to the full sensor under the lock, pulls a settled frame (`_capture_settled` waits for the control to take effect), then restores the previous crop; the momentary widening is a deliberate, brief glitch in the live feed while configuring
- `get_state()` — returns the crop as sensor pixels + normalized box + sensor dimensions (for the UI)

**`birdscanner/detector/pipeline/track_logging.py`** — the `tracking` logger module: `configure_logging(debug)` sets up the stdout stream handler at DEBUG/INFO (called once from `main`), and `TrackingLogger(labels=None)` logs stable-track and track-deletion events (used by `gating.build_gating`, which passes it the COCO `get_labels(intrinsics)` list). These event lines are deliberately terse — just `track_id`/`stable_frames` (+ `missing_frames` on delete); the box is **not** logged (noise) and the classified species is **not** logged here (it is `None` at these points anyway). The **deleted** line additionally carries `class=<object-detection label>` — the COCO class the track represented (e.g. `bird`/`person`/`cat`), resolved from the track's stored `category` index via `detection_utils.label_for_category` (falling back to `id:<n>` for an out-of-range/unlabelled category, or `unknown` when no category was recorded). The category is threaded onto each `StableTrack` by `StableDetectionTracker.update_frame` (which reads `detection.category` off the per-frame detections; `update_tracks_for_frame` itself stays purely box-based), so the log finally says *what kind of thing* a short-lived/never-classified track was. The classified species is instead logged once per saved detection by `classification_pipeline._persist_detection` as a `Bird classified: track_id=… species=… confidence=…%` line followed by a `Saved to <path>` line, so a species only appears in the log when a stable track is actually classified as a bird and persisted

### Bounding box format

All boxes throughout the codebase are `(x, y, w, h)` in ISP output pixel coordinates after `imx500.convert_inference_coords`. The `preprocess_roi` function expands the box to a square with 20% padding before passing to the classifier.

### Colour channel order (RGB vs BGR)

The `main` ISP stream is configured with `"format": "BGR888"` — **not** `"RGB888"`. picamera2's `888` format names are byte-reversed relative to the numpy array they produce: `"BGR888"` yields an `[R, G, B]`-ordered array, while `"RGB888"` yields `[B, G, R]`. Everything downstream assumes **RGB**: the ConvNeXt classifier (ImageNet-RGB normalisation in `build_preprocessing`), `save_thumbnail` (`PIL.Image.fromarray`), and the `cv2.cvtColor(..., RGB2BGR)` writes in `classification_pipeline.py` / `camera_server.py`. Requesting `"RGB888"` therefore swaps red↔blue in every saved image **and** feeds the classifier mis-ordered channels, producing garbage species predictions. If colours ever invert again, check this format string first.

## Conventions

- All functions must have type hints and docstrings.
- The codebase is one package, `birdscanner`, with absolute imports; the layering `detector → ml → db` / `api → db` is one-way (`ml/` never imports `detector/` or `api/`).
- Runtime code lives under `birdscanner/`; dev-only code (never imported by the services, excluded from the Docker images) lives under `tools/` (offline builders/scripts) and `dev/` (the off-Pi camera emulator). `dev/` may import `birdscanner`, but `birdscanner` must never import `dev/` or `tools/`.
- High-confidence classified bird images are written to `$IMAGE_DIR/{species}/` (env var, defaults to `/home/stefan/Pictures/bird_detections`). A 200×200 JPEG thumbnail is saved alongside each image with a `_thumb.jpg` suffix.
- `birdscanner/db/` tests use SQLAlchemy `StaticPool` to share an in-memory SQLite connection across threads (via the shared `engine` fixture in `tests/conftest.py`).
- `birdscanner/api/` tests (under ``tests/api/``) override FastAPI dependencies via ``app.dependency_overrides`` (the ``make_client``/``client`` fixtures in ``tests/api/conftest.py``) so no real DB or filesystem is needed.

### Notes
- `.env` must be created from `.env.example` before the first run (it is git-ignored).
- The `data` Docker volume is the single source of truth: the `detector` service writes images, the per-detection **mp4 clips** (alongside the stills under `$IMAGE_DIR/{species}/`), the SQLite DB, the crop-region JSON (`CROP_CONFIG_PATH`, default `/data/crop.json`), and the **runtime settings overlay** (`SETTINGS_PATH`, default `/data/settings.json`); the `api` service mounts it read-only.
- **Runtime settings** (the Settings page) flow like the crop config: the API never writes `settings.json`; it `POST`s changes to the detector's control server, which validates/persists them and applies the **live** fields immediately. **Restart-only** fields take effect on the next detector boot — the page's "Apply & restart detector" button hits `POST /api/settings/restart`, and the detector exits so the compose `restart: unless-stopped` policy relaunches it (which re-reads `settings.json` at startup). If you change the **save location** (`image_dir`) via the UI, keep it under the shared data volume and set the `api` service's `IMAGE_DIR` env to match, or the gallery won't find the new images. The crop config flows the other way too — the API never writes the file; it `POST`s crop changes to the detector's HTTP server, which applies them live and persists the JSON.
- **Video encoding is software** (the Pi 5 has no hardware encoder) and uses the apt `python3-opencv` FFmpeg videoio backend to encode **H.264 (`avc1`)** — the codec browser `<video>` elements require. Debian's `libavcodec` (pulled in by `python3-opencv` in `Dockerfile.detector`) is built with libx264, so `avc1` is available with no extra package. If the writer can't open an H.264 encoder it falls back to `mp4v` (logged as a warning); an `mp4v` clip is a valid file but is **not browser-playable** — the Lightbox `<video>` shows only the poster (thumbnail), which looks like a single frozen frame. If clips fail to write (empty/absent mp4s), check that the opencv build has videoio/FFmpeg support; if a clip saves but won't play in the browser, check the detector log for the `avc1`-unavailable fallback warning.
- The **species-reference bank** also rides on the data volume at `/data/species_reference` (`SPECIES_REFERENCE_DIR`). It is built offline (`tools/build_species_reference.py`) and **not** baked into the api image — copy `manifest.json` + `images/` into `<data-volume>/species_reference/` on the host, **or** point the builder straight at the deployed bank with `--output-dir <data-volume>/species_reference` (defaults to `$SPECIES_REFERENCE_DIR`). Either way, **restart the api** afterwards so it re-reads the manifest (memoized per-directory). Neither service writes it at runtime; the api just reads it (read-only mount). When absent, the Lightbox reference panel 404s for every species ("No reference available"); when present but thumbnail-less, the tiny "In season" icons fall back to full-res originals — the sign the deployed bank predates the thumbnail backfill.
- `privileged: true` is scoped to `detector` only (required for IMX500 camera device access).
- The `detector` `expose`s port 8000 (the control server from `birdscanner/detector/hardware/camera_server.py`) on the internal compose network only — it is **not** published to the host. The `api` reaches it via `DETECTOR_URL` (default `http://detector:8000`); the detector binds the port from `CAMERA_SERVER_PORT` (default 8000). Both vars live in `.env`/`.env.example` and the defaults work as-is — only change them together. This powers the Camera tab's "Test Camera" button, the detection-region editor (`/crop`, `/capture/full`), **and** detection deletes (`DELETE /api/detections/{id}` proxies to `${DETECTOR_URL}/detections/{id}`).
- The `detector` image (`Dockerfile.detector`) is based on `dtcooper/raspberrypi-os:bookworm` so the system `python3` can import the apt-installed `python3-picamera2` / `python3-libcamera` bindings (these are built natively against the Pi's libcamera and are **not** on PyPI). `numpy`, `opencv` and `pillow` are installed via apt (`python3-numpy` / `python3-opencv` / `python3-pil`); the apt `numpy` is `1.24.2`, which is what the apt-built picamera2/simplejpeg/opencv stack is compiled against. `requirements.detector.txt` therefore pins `numpy==1.24.2` (so pip does not pull numpy 2.x into `/usr/local` and shadow the apt copy — that crashes the detector with `numpy.dtype size changed`) and `onnxruntime==1.23.2` (the version known to run against numpy 1.24.2 on this Pi). These plus `sqlmodel` are pip-installed into the system interpreter with `--break-system-packages`. The IMX500 firmware + `.rpk` network models (`/usr/share/imx500-models/...`) come from the `imx500-all` apt package baked into the image. A plain `python:3.x` base will crash the detector with `ModuleNotFoundError: No module named 'libcamera'`.
- The `detector` service additionally mounts `/run/udev:ro` (libcamera enumerates cameras via udev), the `/dev/dma_heap` device (picamera2 buffer allocation), `/sys/kernel/debug` (debugfs), and `/usr/share/imx500-models:ro` — all required for the camera to initialise inside the container. The `imx500-models` mount exposes the **host's** `.rpk` network firmware to the container: the image's `imx500-all` apt package does not ship the `imx500_network_yolo11n_pp.rpk` model the detector loads (see `birdscanner/detector/config/config.py`), so without the mount the detector logs `Camera not available (Firmware file ...yolo11n_pp.rpk does not exist.)` and retry-loops forever. Mounting the host directory (which has the model, since the bare detector runs there) keeps the container in sync with the host's apt package without baking the binary into the image. The debugfs mount is needed because `IMX500.__init__` opens `/sys/kernel/debug/imx500-fw:<id>/fw_progress` to track on-chip firmware upload; Docker does not expose debugfs to containers (even privileged ones) by default, so without this mount the detector crashes with `FileNotFoundError: ... /sys/kernel/debug/imx500-fw:11-001a/fw_progress`.
- `main.py` configures the camera with `buffer_count=6` (not the bare-Pi example's `12`). Inside the container the kernel CMA / dma-heap pool is host-global (Docker memory limits don't apply to it) and is shared with the IMX500 firmware upload and the always-allocated 2028×1520 raw sensor stream (~3 MB/buffer, the dominant consumer — it's fixed at the sensor's smallest mode and allocated by the Pi 5 PiSP pipeline regardless of config, so it can't be shrunk). With 12 buffers this exhausted CMA and crashed `picam2.start()` with `OSError: [Errno 12] Cannot allocate memory` at the dma-heap `alloc` ioctl. 6 buffers halves DMA pressure while keeping jitter margin; the `main` stream is sized so its longer edge is `DEFAULT_LONG_SIDE=1280` (square for a square crop, smaller on the short side for a non-square crop — see `main_stream_size_for_crop`). This was bumped from 640 to sharpen the still/clip, so the `main` buffers are now ~4× the old 640×640 footprint; the offsetting saving is that the default cropped-clip mode no longer requests the unpacked raw stream (~−14 MB, see `build_camera`). The ConvNeXt classifier crops bird ROIs from `main` (the on-chip detector runs at a fixed 320×320 baked into the `.rpk`, independent of the streams). If 6 ENOMEMs after the 1280 bump, drop `DEFAULT_LONG_SIDE` back toward 640 or increase the host CMA pool in `/boot/firmware/config.txt` rather than dropping `buffer_count`.
- Model files (`assets/models/convnext_v2_tiny_int8.onnx` and `assets/models/BirdNET+_Geomodel_V3.0.3_Global_12K_FP32.onnx`) must be present in the build context on the Pi; they are baked into the image at build time (not shipped in the repo — the `.onnx` binaries are out-of-band).
- The detector runs as `python3 -m birdscanner.detector.main` from `WORKDIR /app`. It loads its data files **package-relative** via `birdscanner/detector/paths.py` (anchored at the package location, resolving to `/app/assets/...` — no longer cwd-relative), so the Dockerfile copies the code to `/app/birdscanner/` and the assets to `/app/assets/` (labels under `assets/labels/`, the classifier ONNX + its class map under `assets/models/`). Two models are copied from `assets/models/`: the classifier `convnext_v2_tiny_int8.onnx` and the BirdNET geomodel `BirdNET+_Geomodel_V3.0.3_Global_12K_FP32.onnx` (used by `birdscanner/detector/pipeline/geo_priors.py` to build the spatio-temporal species prior at startup) — any other model files there are training/conversion scratch and are **not** copied. Each is `COPY`d on its own line before the source layers so Docker caches the large binaries across code edits. If the classifier is missing, onnxruntime fails at startup with `NO_SUCHFILE: Load model from .../convnext_v2_tiny_int8.onnx failed`; if the geomodel is missing the detector does not crash — `geo_priors.refresh_geo_priors` logs `Could not build geo prior (... NO_SUCHFILE ...)` and runs on without a location prior.
- If the camera is unavailable (dev-node missing — unplugged, mis-seated, or device access not granted), the `detector` does **not** crash-loop: `wait_for_camera()` logs `Camera not available (...). Retrying in 30s...` and retries until the camera appears. The `api` service is independent and continues serving the site; because the detector initialises the DB schema on startup, the UI loads (showing existing images, or an empty gallery on a fresh volume).
